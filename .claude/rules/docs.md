docs/ の作成・更新ルール

## ファイルの種類（サフィックスで区別）

| サフィックス | 役割                                                          | 例                         |
| ------------ | ------------------------------------------------------------- | -------------------------- |
| `.md`        | 仕様書。確定した仕様                                          | `authentication.md`        |
| `.plan.md`   | 計画メモ。設計案・フェーズ分け。実装後に仕様書へ昇格 or 削除  | `content-import.plan.md`   |
| `.report.md` | 調査レポート。外部API等の技術調査。コードに吸収されたら削除可 | `bunny-api.report.md`      |
| `.repo.md`   | リポジトリ参考資料。既存システム分析、技術比較                | `current-analysis.repo.md` |

迷ったら: 確定した仕様か → `.md`、まだ提案段階か → `.plan.md`、外部の技術情報か → `.report.md`

## 仕様書（.md）の構成

以下のセクションで構成する:

1. **Overview** — 2〜4行の箇条書き。「何であるか」を宣言的に
2. **Related Docs** — 他ドキュメントへの相互参照。関連ドキュメントがなければ省略可
3. **Specification** — 本体。Data Model / API / Page 構成など
4. **Decision Log** — 設計判断の記録（背景・候補・決定・理由）。末尾に追記する。設計判断がなければ省略可

## 書くこと

- データモデル — フィールド定義、リレーション
- API 仕様 — HTTPメソッド、パス、リクエスト/レスポンスの型
- ページ/URL 構造 — URLパターン、UI構成要素
- 判断の根拠 — 代替案を含めた比較と選定理由。覆した判断も消さず注釈付きで残す
- スコープ境界 — 「やらないこと」「後でやること」を明示

## 書かないこと

- 実装の詳細（関数名、クラス名、内部の呼び出しフロー）
- フレームワークの作法（フレームワーク固有の API 名や規約）
- ビジュアルデザイン（色、余白、タイポグラフィ）
- テストケース
- デプロイ/インフラ
