# ai-kit

Claude Code で AI 駆動開発を行うためのテンプレートリポジトリ

## 思想

- **ドキュメントが信頼の源泉**
  - コードを書く前に仕様書で合意する。AI はドキュメントを起点に実装する
- **役割の分離**
  - 設計・実装・レビューを別々の AI インスタンスで行い、視点の汚染を防ぐ
- **失敗を資産にする**
  - 障害・失敗を記録し、パターンが見えたらルールに昇格させる


## 3AI 駆動方式

人間と対話する AI は1つだけ。実装やレビューは独立したサブエージェントが行う

| 役割           | 実行方式         | 知っていること          | 知らないこと        |
| -------------- | ---------------- | ----------------------- | ------------------- |
| 開発パートナー | 人間と対話       | 議論経緯、docs/、otuge/ | 実装の試行錯誤      |
| 実装 AI        | サブエージェント | design.md, plan.md      | 設計時の議論        |
| 説明 AI        | サブエージェント | 実装コードのみ          | design.md、設計意図 |

詳細: [`.claude/skills/miko/three-ai-method.md`](.claude/skills/miko/three-ai-method.md)


## 開発フロー

### 日常の開発

1. `/miko {やりたいこと}` で開発を始める
2. `otuge/{タスク}/design.md` が完成するまで議論を深める
3. AI がプラン → 実装 → 説明照合 → 設計照合 → コードレビューまでを自動で回す
4. 動作確認等問題なければ、１回目のコミットを行う（ `otuge/` も含めてコミットする）
5. 最後に `docs/` に反映し、otuge を削除し、２回目のコミットを行う

> [!IMPORTANT]
> `docs/` の更新は必ず最後に行うこと  
> 実装漏れ、ドキュメントの更新漏れを減らすためです

### 判断に迷った時

`/tengi` で複数 AI に議論させ、論点を整理してから人間が判断する

> [!TIP]
> 必ず「エージェントチーム」機能を有効にしておくこと

> [!NOTE]
> `adversary` スキルは Codex にする予定。Codex は反例を見つけるのが上手らしい

### 簡易修正

スタイル調整や1-2ファイルの変更は `/miko` がフローを回さず直接実行する

### ハマった時・ミスした時

`.claude/incidents.md` に記録する。原因・対策・教訓を残すことで、同じ問題を繰り返さない  
パターンが溜まったら `/incidents-review` でルールに昇格させる


## スキル一覧

### /miko — 開発パートナー（御子）

メインの対話相手。設計からレビューまでの6フェーズを統率する:

1. **設計** — 人間と対話して `otuge/{タスク}/design.md` を作成
2. **プラン作成** — 実装 AI にプランを作らせ、設計と照合
3. **実装** — 実装 AI にコードを書かせる
4. **説明照合** — 説明 AI にコードを説明させ、設計と照合
5. **設計照合** — 設計照合 AI がレビュー
6. **コードレビュー** — 品質・セキュリティチェック

簡易修正（1-2ファイルの変更）はフローを回さず直接実行する

### /tengi — 天議

設計判断に迷った時に、複数の AI エージェントで議論する仕組み

- 3人 or 5人のメンバーを並列起動
- 各メンバーが異なる観点（実装コスト、保守性、UX 等）で評価
- 反例検出 AI が事前に穴を洗い出す
- 意見を集約して改善項目リストを生成

モード:
- **agent**: メンバーを再起動して議論（重要な判断向け）
- **solo**: 進行役が1人で突き合わせ（軽い議題向け）

### その他のスキル

| スキル              | 用途                                 |
| ------------------- | ------------------------------------ |
| `/implement`        | 設計ドキュメントからプラン作成・実装 |
| `/explain`          | 設計意図なしでコードを説明           |
| `/design-review`    | 設計とコードの照合レビュー           |
| `/code-review`      | git diff ベースのコードレビュー      |
| `/adversary`        | 議題・方針の穴を検出                 |
| `/incidents-review` | 障害記録の棚卸し・ルール昇格         |

## 検証中の仕組み

### Phase 4（説明照合）と Phase 5（設計照合）の重複

この2つのフェーズは意図的に似た検証を行っている

- **Phase 4（説明照合）**: 説明 AI がコードを読み、「何をしているか」を説明 → 設計と照合
- **Phase 5（設計照合）**: 設計照合 AI が design.md と explanation.md を直接照合

どちらの方が設計との乖離を検出しやすいかを比較検証している

検出した問題は `docs/reference/explain-effectiveness.repo.md` に記録する  
将来的にどちらか一方に統合する可能性がある
